> 本文档定义 Agent 的核心循环（Loop）引擎，包括自查、复盘、Subagent 会诊和自评估迭代优化四大机制。

## 二、自查要求

### 2.1 每轮 Loop 开始前自查（Loop Pre-Check）

在进入新一轮采集前，Agent 必须逐项检查：

| 检查项 | 检查方法 | 不通过处理 |
|--------|---------|-----------|
| **剩余 gap 数量** | `len(gaps) == 0` → 跳过所有查询 | 如果 gaps 为空，直接 break |
| **当前 loop 计数** | `state.loop >= MAX_LOOP` | 如果已达上限，不再进入循环体 |
| **上一轮修复数** | `fixed == 0` 且 `len(gaps) > 0` | 标记 WARN：「上一轮零修复，可能数据源异常」 |
| **错误累积趋势** | `state.errors` 连续 2 轮递增 | 标记 WARN：「错误持续增加，建议检查 FlyAI 服务」 |

```python
# 自查伪代码（对应于 agent_v5.py collect_data() 中的循环控制）
if not gaps:                          # 自查1: 缺口已清空
    print("✅ 所有数据已获取，提前退出")
    break
if loop == MAX_LOOP:                   # 自查2: 已达上限
    state.add_warning("Loop", f"已达上限 {MAX_LOOP} 轮，强制退出")
```

### 2.2 单项数据采集后自查（Item Post-Check）

每完成一个数据项采集，立即执行：

```python
# 酒店数据深度自查
hotels = get_items(results.get("hotel", {}))
if hotels:
    # 自查a: 数量是否满足三方案需求
    if len(hotels) < 3:
        state.add_warning("酒店", f"仅 {len(hotels)} 条，不满足三方案最低要求")
    # 自查b: 价格字段是否可解析
    valid_prices = [h for h in hotels if price_parse(h.get('price','')) < 99999]
    if len(valid_prices) < len(hotels):
        state.add_warning("酒店", f"{len(hotels)-len(valid_prices)} 条价格无法解析")
    # 自查c: 名称字段是否非空
    unnamed = [h for h in hotels if not h.get('name')]
    if unnamed:
        state.add_warning("酒店", f"{len(unnamed)} 条缺少酒店名")
```

### 2.3 自查清单（Checklist）

每轮 Loop 结束时输出：

```
─────────────────────────────────────────
  第 {N} 轮自查报告
─────────────────────────────────────────
  ✅ 数据完整性: {M}/4 项成功
  ✅ 价格有效性: {P}/{total} 条可解析
  ✅ 名称完整性: {Q}/{total} 条有名称
  🚗 自驾语义:   ✅ 已区分 | ⚠️ 歧义(默认当地租车)
  ⚠️ 剩余缺口:   {gaps_remaining}
  ⚠️ 降级次数:   {downgrade_count}
  ❌ 本轮错误:   {error_count}
─────────────────────────────────────────
```

### 2.4 🚗 自驾语义校验（专项自查）

> 在 parse() 解析完成后，Loop 开始前，必须执行此项校验。
> 详见 [自驾语义约束.md](自驾语义约束.md)

```python
# 自驾语义自查伪代码
def check_drive_semantics(info: dict) -> str:
    if "自驾" not in original_query and "开车" not in original_query:
        return "N/A — 无关"
    
    if info.get("drive_mode") == "none":
        # 严重错误：用户说了自驾但没正确解析
        state.add_error("自驾语义", "用户输入包含自驾关键词但 drive_mode=none，解析逻辑有 bug")
        return "❌ 未区分"
    
    if info.get("drive_mode") == "当地租车" and "歧义" in str(state.warnings):
        return "⚠️ 歧义（默认当地租车）"
    
    return f"✅ {info['drive_mode']}"
```

**自查要点**：

| 用户说了什么 | 预期 drive_mode | 如果不一致 |
|-------------|----------------|----------|
| "自驾" 但没说"全程""租车" | `当地租车` + ⚠️ | 全程自驾 → 费用偏差巨大（¥1,920 vs ¥2,150） |
| "全程自驾" / "开车过去" | `全程自驾` | 当地租车 → 缺少长途费用估计 |
| "当地租车" / "到了租车" | `当地租车` | 全程自驾 → 多算了不存在的长途费 |
| "对比两者" | `both` | 只算一个 → 缺失对比 |

## 三、复盘要求

### 3.1 每轮复盘（Per-Loop Review）

每轮 Loop 结束后，在终端输出结构化复盘摘要：

```
  Loop {N}: ✅🏨 ✅🚄 ⏳🎯 ⏳✈️ | 修复 2 | 剩余缺口 2

  复盘:
  ✅ 成功: hotel(L{N}), train(L{N})  — 主策略一次命中
  ⏳ 失败: poi, flight                — 主策略+降级均失败
  💡 建议: poi/flight 连续 {K} 轮失败，建议下轮跳过以节省时间
```

**复盘必答三问**：
1. **为什么成功？** 成功的数据项用了哪种策略（主策略/降级）？原因是什么？
2. **为什么失败？** 失败的数据项出现什么错误（超时/JSON解析/退出码非0）？是否可复现？
3. **下一步？** 下一轮应该改变什么策略？是否应该跳过某项？

### 3.2 终局复盘（End-to-End Review）

全部 Loop 结束后，在方案文件底部生成终局复盘：

```markdown
## 采集复盘

| 数据项 | 最终状态 | 尝试次数 | 成功策略 | 失败原因 |
|--------|---------|---------|---------|---------|
| 🏨 酒店 | ✅/❌ | {N} | 主策略(L1) / 降级 | / |
| 🎯 景点 | ✅/❌ | {N} | / | 超时×3 |
| 🚄 火车 | ✅/❌ | {N} | 降级(L2) | 主策略参数错误 |
| ✈️ 机票 | ✅/❌ | {N} | / | 全策略失败 |

### 效率分析

- 总耗时: {elapsed}s
- 有效查询: {success_count} 次
- 无效查询: {fail_count} 次（浪费 {waste_time}s）
- 一次命中率: {success_count}/{total_queries} = {rate}%
```

### 3.3 复盘档案要求

每次运行结束后，将复盘摘要追加到 `复盘记录.md`（按日期归档）：

```markdown
## 2026-07-05 大理5天 复盘

- 查询: "一家三口大理5天4夜 预算2万"
- 结果: 0/4 成功 | 6 轮循环 | 134s
- 根因: FlyAI CLI Windows 兼容性问题 (UV_HANDLE_CLOSING)
- 建议: 迁移到 Linux 环境运行
- 修复: 待 FlyAI 版本更新
```

## 四、调用 Subagent 会诊要求

### 4.1 触发条件

当出现以下任一情况时，Agent 应**暂停 Loop**，触发 Subagent 会诊：

| 触发条件 | 阈值 | 会诊目的 |
|----------|------|---------|
| 连续失败 | 同一数据项连续 2 轮全策略失败 | 确认是否为永久性故障，是否需要更换数据源 |
| 数据质量低 | 获取的数据条数 < 3 且价格全部不可解析 | 判断是否需要换关键词/换城市 |
| 全部缺口 | 2 轮后 `len(gaps) == 4`（没有任何进展） | 诊断是否是环境问题（网络/服务/兼容性） |
| 方案质量差 | 生成的 `hotels < 3` 无法输出三方案对比 | 是否需要降级到「仅提供建议不上具体酒店」模式 |
| 错误激增 | `state.errors` 数量每轮翻倍 | 是否存在雪崩效应，需要中断 |

### 4.2 会诊 Subagent 类型与分工

| Subagent | 专长 | 调用时机 | 输出 |
|----------|------|---------|------|
| **数据诊断 Agent** | 分析 API 调用日志 | 连续失败时 | 失败根因 + 修复建议 |
| **策略优化 Agent** | 替换搜索策略/关键词 | 数据质量低时 | 新策略列表（≥3个备选） |
| **环境诊断 Agent** | 检查运行环境 | 全部缺口时 | 环境问题诊断报告 |
| **方案质量 Agent** | 审核输出方案 | 方案质量差时 | 质量评分 + 改进建议 |
| **熔断决策 Agent** | 判断是否中止 | 错误激增时 | 继续/中止 决策 |

### 4.3 会诊流程

```
Loop 异常触发
    │
    ├──► ① 收集上下文快照:
    │       - state 当前状态
    │       - 最近 2 轮日志
    │       - 失败项的错误详情
    │
    ├──► ② 选择 Subagent:
    │       根据触发条件匹配最合适的 Subagent
    │
    ├──► ③ 发送会诊请求:
    │       {
    │         "trigger": "consecutive_failure",
    │         "context": { ... },
    │         "target": "flight",
    │         "attempts": 3,
    │         "all_errors": [ ... ]
    │       }
    │
    ├──► ④ 接收会诊建议:
    │       Subagent 返回诊断 + 策略建议
    │
    └──► ⑤ 执行建议:
            - 如果是策略建议 → 替换下一轮策略
            - 如果是中止建议 → 记录原因，优雅退出
            - 如果是环境问题 → 输出诊断报告，标注 ⚠️
```

### 4.4 会诊记录格式

每次会诊结果写入 `会诊记录.md`：

```markdown
## 会诊 #1 — 2026-07-05 07:18

- **触发条件**: 全部缺口 (2轮后 0/4)
- **调用 Subagent**: 环境诊断 Agent
- **诊断结果**: FlyAI CLI Windows 兼容性 bug (UV_HANDLE_CLOSING)
- **建议**: 迁移到 Linux 环境 / 等待 FlyAI 版本更新
- **决策**: 继续 Loop，输出错误报告
- **影响**: 方案仅含自驾数据 + ⚠️ 标注，不含实时酒店/机票
```

## 五、自评估迭代优化要求

### 5.1 方案生成后自评估（Post-Generation Self-Assessment）

方案生成后，Agent 必须对自身输出进行打分：

```python
def self_assess(md_content: str, info: dict, results: dict, gaps: list) -> dict:
    """Agent 自评估"""
    scores = {
        "数据完整度": 0,    # 0~25 分
        "酒店方案质量": 0,   # 0~30 分
        "行程丰富度": 0,     # 0~20 分
        "预算合理性": 0,     # 0~15 分
        "可读性": 0,        # 0~10 分
        "总分": 0           # 0~100 分
    }
    
    # ── 数据完整度 (25分) ──
    acquired = 4 - len(gaps)
    scores["数据完整度"] = acquired * 6 + (1 if acquired == 4 else 0)  # 满分25
    if len(gaps) > 0:
        scores["数据完整度"] -= len(gaps) * 3  # 每个缺口扣3分
    
    # ── 酒店方案质量 (30分) ──
    hotels = get_items(results.get("hotel", {}))
    if len(hotels) >= 3:
        scores["酒店方案质量"] = 25
        # 三档齐全 +5
        prices = [price_parse(h.get('price', '')) for h in hotels]
        tiers = set(hotel_tier_name(p) for p in prices)
        if len(tiers) >= 3:
            scores["酒店方案质量"] += 5
    elif len(hotels) > 0:
        scores["酒店方案质量"] = 8 * len(hotels)  # 每有一个得8分
    # 0个酒店 → 0分
    
    # ── 行程丰富度 (20分) ──
    pois = get_items(results.get("poi", {}))
    scores["行程丰富度"] = min(20, len(pois) * 4)  # 每个POI 4分，满分20
    
    # ── 预算合理性 (15分) ──
    if "✅ 预算内" in md_content:
        scores["预算合理性"] = 15
    elif "⚠️ 超" in md_content:
        scores["预算合理性"] = 8  # 超预算扣分
    else:
        scores["预算合理性"] = 5  # 无预算评估
    
    # ── 可读性 (10分) ──
    readability_checks = [
        "## 一、" in md_content,     # 有章节结构
        "## 二、" in md_content,
        "| " in md_content,          # 有表格
        "✅" in md_content,          # 有状态标记
        "⚠️" in md_content,
    ]
    scores["可读性"] = sum(2 for c in readability_checks if c)
    
    scores["总分"] = sum(v for k, v in scores.items() if k != "总分")
    return scores
```

### 5.2 自评估等级

| 总分 | 等级 | 含义 | 后续行动 |
|------|------|------|---------|
| 90~100 | 🏆 优秀 | 数据完整、三方案齐全、预算合理 | 直接交付 |
| 70~89 | ✅ 良好 | 有小缺陷但可用 | 标注改进建议，交付 |
| 50~69 | ⚠️ 及格 | 数据缺失较多，方案可用性一般 | **触发 Subagent 会诊** |
| 0~49 | ❌ 不合格 | 严重缺失，建议不交付 | **必须重新生成或人工介入** |

### 5.3 迭代优化决策树

```
自评估完成
    │
    ├── 等级 ≥ 70 (✅ 良好)
    │   └──► 直接交付，附改进建议
    │
    ├── 等级 50~69 (⚠️ 及格)
    │   └──► 触发会诊 → 尝试修复 → 重新生成
    │        ├── 修复成功 → 等级提升 → 交付
    │        └── 修复失败 → 标注限制，降级交付
    │
    └── 等级 < 50 (❌ 不合格)
        └──► 检查是否可重试:
             ├── 环境问题（如 FlyAI 崩溃）→ 更换环境重试
             ├── 参数问题 → 调整 query 重试
             └── 永久性故障 → 输出错误报告，人工介入
```

### 5.4 连续迭代追踪

### 5.5 优化规则引擎

Agent 根据自评估结果自动生成优化规则，供下次执行时参考：

```python
def generate_optimization_rules(scores: dict) -> list:
    """根据自评估生成优化建议"""
    rules = []
    
    if scores["数据完整度"] < 20:
        rules.append("RULE-001: 优先检查 FlyAI 服务可用性")
        rules.append("RULE-002: 若 2 轮内无任何数据，先执行环境诊断")
    
    if scores["酒店方案质量"] < 20:
        rules.append("RULE-003: 酒店降级策略增加「携程搜索」关键词")
        rules.append("RULE-004: 尝试放宽 check-in-date 到 ±3 天范围")
    
    if scores["行程丰富度"] < 10:
        rules.append("RULE-005: 景点主策略失败后尝试「{目的地}亲子游攻略」关键词")
    
    return rules
```
