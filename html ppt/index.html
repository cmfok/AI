大家好，今天分享的主题叫「AI 真垃圾」。这不是标题党——是我过去半年用 AI 的真实感受。从觉得这玩意儿就是个高级搜索引擎，到现在每天离不开它，中间踩了无数的坑。今天就把这些坑和填坑的方法分享给大家。

Problem · 第一个认知冲击

不学它的语言，它就不理你

AI 不是搜索引擎。它是外星人——你得学会说它的语言。

🔍 搜索引擎思维

  • 输入关键词 → 返回链接
  • 越短越好，精准检索
  • 不会就换关键词再搜

🤖 AI 对话思维

  • 输入自然语言 → 直接生成答案
  • 越详细越好，给出上下文
  • 不会就教它、纠正它、迭代它
大部分人说「AI 不行」，其实是用搜索引擎的习惯在用 AI。输入三个关键词，期望得到一个精准答案——这不现实。AI 需要你给出完整上下文、明确角色、说明格式要求。

Training · 大模型是怎么学会说话的

训练三阶段：预训练 → 微调 → RLHF

理解这三个阶段，你就知道为什么有时候 AI「很聪明」、有时候「很蠢」。

1

预训练 Pre-training

海量互联网文本喂给模型，学会语法、常识、推理模式。这一阶段决定模型的「知识天花板」。

知识基础
2

微调 Fine-tuning

用高质量人工标注数据教模型特定任务。让它学会「什么样的回答才是好的」。

任务对齐
3

RLHF 人类反馈强化

人类对多个回答打分排名，模型学习人类偏好。拒绝有害请求、遵循指令格式。

价值观对齐
预训练像通识教育——什么都学过但什么都不精；微调像职业培训——针对具体任务优化；RLHF 像导师指导——教它什么是对的、什么是错的。三个缺一不可。

Truth · 诚实面对

AI 能做什么、不能做什么

✓ 擅长的事

  • 文本总结 · 翻译 · 改写
  • 代码生成 · Bug 修复 · 重构
  • 创意写作 · 头脑风暴
  • 数据分析 · 表格处理
  • 格式转换（JSON→Markdown 等）

✗ 不擅长的事

  • 精确数学计算（需要工具辅助）
  • 实时信息（知识截止训练日期）
  • 理解「说不清的需求」
  • 做需要物理世界常识的决策
  • 「一次做对」——它需要迭代
核心矛盾：AI 的强项是「理解模糊指令」和「快速生成」，弱项是「精确计算」和「实时知识」。所以最佳实践是：让 AI 做它擅长的，用工具补它不擅长的。

Choose · 怎么选模型

模型选型：不是越贵越好

❌ 错误做法

  • 所有任务都用 GPT-4o
  • 不看价格，只看排行榜
  • 一套 prompt 打天下
  • 忽略上下文窗口限制
  • 不利用模型各自优势
→

✅ 正确做法

  • 简单任务 → DeepSeek 足够
  • 看任务需求，不看排行榜
  • 不同模型配不同 prompt 策略
  • 超长文档 → Claude / Gemini
  • 组合用：DeepSeek 起草 + GPT-4o 润色
就像你不会用小刀砍树、用斧头削苹果。选模型也一样的道理——看任务、看预算、看上下文长度，合理分配。

Case 1 · 实战案例

从「老板一通电话」到「一份完整 PPT」

📞 输入：一句话需求

「小王，下周给我做个 OKR 管理的方案，要能自动算分、能生成报表、能发飞书通知。」

AI 做了什么

  • 语音 → 文字（飞书妙记）
  • 文字 → 需求结构化
  • 需求 → 技术方案框架
  • 方案 → PPT 大纲 + 逐页内容

📊 输出：结构化的 15 页方案

$ 转写 → 结构化 → 方案
✔ 提取关键需求：自动算分 + 报表 + 飞书通知
✔ 拆解为 5 模块 · 15 页 PPT
✔ 技术方案：MySQL + Python + 飞书 API
✔ 输出：PPT 大纲 + 逐字演讲稿
$ 从电话到方案：20 分钟完成
以前这种事要花半天：整理需求、画架构图、写方案、做 PPT。现在用 AI，20 分钟从语音到成品。关键不是 AI 多聪明，是你知道怎么把模糊需求拆成它能理解的一步步。

Part 2

提示词工程
Prompt Engineering

不是写一句话——是写一封「说明书」

Framework · 好的 Prompt 长什么样

提示词 六要素

  • ① 角色 Role

    「你是一个资深 Python 后端工程师」—— 给它身份，它就按身份说话。

  • ② 任务 Task

    「请分析这份对账数据的差异原因并给出修复方案」—— 明确要干什么。

  • ③ 上下文 Context

    「这是财务部门的月度对账数据，差异率不能超过 0.1%」—— 给背景。

  • ④ 格式 Format

    「以 Markdown 表格输出，包含差异项、原因、修复 SQL」—— 定格式。

  • ⑤ 示例 Example

    「参考这段日志格式：2026-06-01|error|...」—— 给样本。

  • ⑥ 约束 Constraint

    「不要修改原始数据表，只生成只读查询」—— 划红线。

六要素背下来：角色、任务、上下文、格式、示例、约束。每次写 prompt 对着这六点检查，命中率能提升 50% 以上。

Case 2 · 真实翻车现场

同样的任务，不同的 Prompt

🗑️ 垃圾 Prompt

「帮我写个约会文案」

→ AI 输出：一段套话模板，毫无灵魂。
→

✨ 优质 Prompt

「你是一个恋爱顾问。帮我写 3 个版本的约会邀请文案，目标对象是喜欢文艺的女生，约去美术馆看展。风格：自然不油腻、有文化感但不装。每个版本不超过 80 字。」

→ AI 输出：3 版各有风格的高质量文案。
同一个模型，同一个任务，prompt 的质量差异就是结果的差异。左边的 prompt 只给了 7 个字，右边的给了角色+目标+风格+数量+字数限制——这就是提示词工程的威力。

上下文工程
Context Engineering

prompt 只占 20%——剩下的 80% 是喂给 AI 的「知识」

Insight · 为什么 Prompt 好还不够

上下文 = 知识 + 规则 + 示例

AI 不知道你的项目用 MySQL 还是 PostgreSQL，不知道你的数据库有 47 张表，不知道「广州仓」叫「South」——除非你告诉它。

Case 3 · 两个真实案例

有上下文 vs 没上下文

❌ 没上下文——运费计算

「帮我计算这个月所有订单的运费」

→ AI 不知道运费表在哪、不知道计费规则、不知道不同渠道费率不同 → 要么瞎编，要么拒绝。

✅ 有上下文——运费计算

「帮我计算运费。数据字典里有 sf_fee_rate 表，项目手册里记录了计费规则：重量×单价+保价费。」

→ AI 直接生成正确 SQL，一次跑通。

❌ 没上下文——简历分析

「帮我分析这份简历合不合适」

→ AI 不知道评分标准、不知道岗位要求、不知道匹配维度 → 输出主观废话。

✅ 有上下文——简历分析

「按评分标准.json 里的 5 个维度评估，岗位要求参照 JD 文档。」

→ AI 输出量化的五维评分表，可追溯可对比。

同样两个任务，有没有上下文就是两种结果。上下文工程的核心就是把「AI 不知道的信息」系统性地喂给它。

Case 4 · 规训前后的天壤之别

一句需求，两种 AI

😈 没规训的 AI

「帮我做一个生日营销活动」

→ 写了 200 行代码：
· 自己建了新的 customers 表
· 自己发明了生日格式 DD/MM
· 没考虑 GDPR 隐私合规
· SQL 字符串拼接（注入漏洞）
❌ 全部推翻重做
→

😇 规训后的 AI

「帮我做一个生日营销活动」

→ 写对了：
· 查了数据字典用 customers_info 表
· 按规范用 snake_case 命名
· 参数化查询防注入
· 写完了运行 /upload-check 自查
✅ 一次通过质检
同一个需求，同一个 AI，加了一套规训体系之后结果天差地别。没有规训的 AI 是脱缰野马，有规训的 AI 是你的得力助手。

Part 5

循环工程
Loop Engineering

让 AI 不停地跑——定时、触发、监控、进化

Evolution · 从手工到自动化

循环工程的 三级跳

① 当前：手工循环

每天打开 IDE，手动触发 AI 对账 → 看报告 → 手动修复 → 手写日志。人不能离开。

  • 依赖人力触发
  • 耗时 30-60 分钟/天
  • 容易遗漏异常
→

② 演进：半自动

crontab 定时触发 → AI 自动分析 → 生成报告 → 推飞书 → 人工确认修复。人只做决策。

  • 定时自动运行
  • 耗时 5 分钟审核
  • 飞书通知异常
→

③ 终极：全自动

AI 监控 → 自动诊断 → 自动修复 → 自动验证 → 自动写报告。人只在需要时介入。

  • 完全无人值守
  • 人只处理边缘 case
  • 持续自我优化
我们现在的对账循环已经走在 ①→② 的路上。目标是让 AI 自己跑、自己修、自己报告，人只在它搞不定的时候拍板。

Scale · 同样的工具，不同的用法

个人用 AI vs 企业用 AI

🧑‍💻 个人使用

  • 打开 ChatGPT 网页，打一句话
  • 不满意就重新生成
  • 换个模型再试试
  • 没有知识库，每次从头讲
  • 没有版本管理，上次的 prompt 丢了
→

🏢 企业使用（AIAD 模式）

  • IDE 内直接调用 AI，所见即所得
  • 规则 + 知识 + 记忆 三件套常驻
  • prompt 模板可复用可版本管理
  • 代码自动 commit + push
  • 发布走审核流水线、自动部署
个人用 AI 就是「聊天」，企业用 AI 是把 AI 嵌入工作流。前者提升个人效率，后者提升组织效能。

Warning · 必须正视的问题

AI 使用中的 数据安全红线

  • 🔴 绝不让 AI 看到的东西

    客户真实姓名/手机号/身份证 · API 密钥和 Token · 支付密码和银行账号 · 未脱敏的订单数据 · 内部财务未公开报表

  • 🟡 脱敏后可用的数据

    客户 ID 替换为「客户A」· 金额脱敏为区间值 · 手机号替换为「138****1234」· 地址保留省市区去掉门牌号

  • 🟢 安全使用原则

    使用本地部署模型（如 DeepSeek 本地版）· 敏感计算在本地做，AI 只看脱敏后数据 · 定期审查 AI 对话记录 · 所有 AI 生成代码走人工 Review

数据安全不是「AI 会不会偷我数据」的问题，是「我有没有把不该给的数据喂给 AI」的问题。养成脱敏习惯，养成本地部署习惯。
